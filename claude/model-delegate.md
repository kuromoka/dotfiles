# 下位モデル委譲の運用ルール

**目的**: メインモデルのトークン消費量を削減する。実装作業を 1 段下のモデルに委譲し、メインモデルは計画・判断・最終確認に専念する。

## 委譲先の決定

メインモデル（このセッションを動かしているモデル）に応じて委譲先を切り替える：

| メインモデル | 委譲先 | Agent の `model` 値 |
|---|---|---|
| Fable | Opus | `"opus"` |
| Opus | Sonnet 4.6 | `"sonnet"` |

以下、本文中の「下位モデル」は上記で決まる委譲先を指す。

## 自動委譲の判断（メインモデル側で判定。確認不要）

ユーザーから依頼を受けた直後、**実装に着手する前** に以下の基準で判定する。タイミングが重要 — メインモデルがファイルを大量に読んでしまった後では、トークンは既に消費済みで委譲効果が薄い。

### 委譲する（下位モデルに投げる）

以下のいずれかに該当し、かつ「設計判断より作業量が支配的」なタスク：

- **複数ファイルにまたがる機械的な編集**: rename、一括置換、API シグネチャ変更の伝播
- **パターン適用 / ボイラープレート**: 既存モジュールに倣った新規スキャフォールド、CRUD、テスト雛形
- **マイグレーション系**: フォーマット変換、import path 一括書き換え、依存ライブラリの API 移行
- **読み込み量が多いタスク**: 多数のファイルを読まないと判断できないが、判断自体は単純なもの（例: 「全テストファイルから deprecated な API 呼び出しを探して直す」）
- **計画が明確化されたタスク**: メインモデルが plan モードや事前の検討で **編集対象のファイルパスと変更内容を特定済み** のもの。規模が小さくても委譲する（思考はメインモデル、実装は下位モデルという分業）

### 委譲しない（メインモデル自身で対応）

- **設計・アーキテクチャ判断**: 抽象化、モジュール分割、API 設計
- **デバッグ・原因調査**: 結果を見ながら方針を変える対話的な作業
- **コードレビュー・批評**: メインモデルの判断品質が必要
- **計画が固まっていない小規模編集**: 何を直すか曖昧なまま着手する 1〜2 ファイルの軽微な変更（オーバーヘッドが利得を上回る）。ただし編集対象が明確なら規模に関わらず委譲する
- **既にメインモデルが大量にコンテキストをロード済みのタスク**: トークンは既に支払い済みなので委譲利得が小さい
- **ユーザーがメインモデルの応答を期待していると明示しているもの**: 「お前が考えて」「直接聞いている」等

## 委譲フロー

1. **ユーザーへの通知**: 委譲する旨を一言だけ伝える（確認は取らない）。例: 「実装は Opus に委譲します」「実装は Sonnet に委譲します」
2. **`Agent` ツールを呼ぶ**（`model` は上記の表に従う）:
   ```javascript
   Agent({
     subagent_type: "general-purpose",
     model: "opus", // Fable がメインの場合。Opus がメインなら "sonnet"
     description: "<3-5 word task>",
     prompt: "<self-contained briefing>",
   });
   ```
3. **プロンプト設計**（トークン効率最優先）:
   - **下位モデルにファイルを読ませる**。メインモデル側で読んだ内容を prompt に貼り付けない（二重課金になる）
   - ファイルパス・関数名・変更仕様を **明示的かつ具体的に** 渡す
   - 「報告は変更点の要約のみ。ファイル内容のダンプは不要」と返答形式を指定する
   - プロジェクトの規約は CLAUDE.md から自動的に読まれるので明記不要
4. **結果のハンドリング**: 下位モデルからの要約を受け取り、ユーザーに簡潔に伝える。差分の再読み込みは検証が必要なときだけ。

## 安全ルール（委譲しても緩めない）

- **破壊的・不可逆な操作** はユーザー確認を挟む（force push、`git reset --hard`、ファイル一括削除等）。下位モデルに委譲する場合も、Agent プロンプトに「destructive な操作は実行せずメインモデルに確認を返す」と明記する
- **外部 API 呼び出し・メッセージ送信** は委譲しない（Slack、メール、PR コメント、デプロイ等）
- **シークレット・認証情報を扱うタスク** は委譲しない

## Dynamic Workflows での委譲

`Workflow` ツール（Dynamic Workflows）はメインモデルで回すとトークン消費が大きい。トークンの大半は「ファイルを大量に読む調査エージェント」が消費するので、**そこを下位モデルに固定する**。

- **調査・読み込み主体のエージェントは下位モデルに落とす**: `agent()` の `model` オプションで指定する（指定しなければメインモデルを継承してしまう）。
  ```javascript
  // 調査フェーズを下位モデルに委譲（Opus がメインなら "sonnet"）
  await agent("このサブシステムを調べて要約を返して", { model: "sonnet", schema: FINDINGS_SCHEMA })
  // フェーズ一括なら map で一律指定
  await parallel(targets.map(t => () => agent(t.prompt, { model: "sonnet", phase: "調査" })))
  ```
- **オーケストレーション（メインループ）はメインモデルのまま**でよい。エージェントを撒いて結果を統合するだけで軽量。重い読み込みは下位モデル側で行わせる。
- **機械的な調査ステージは `effort: "low"` も併用**してさらに安くする。
- **サブエージェントには要約 / 構造化結果を返させる**（`schema` を使う）。生ファイルをメインループに戻すと二重課金になる。
- 判断基準は上記「自動委譲の判断」と同じ。**設計・原因調査・レビューなど判断品質が要る部分はメインモデル**のエージェント（または統合段）に残す。

## codex 委譲との使い分け

| シグナル | 委譲先 |
|---|---|
| ユーザーが「codex に」「rescue で」と明示 | `codex:codex-rescue` |
| ユーザーが usage limit に近いと発言 | `codex:codex-rescue`（提案あり） |
| 上記以外で、実装量が多く機械的なタスク | `general-purpose` + 下位モデル（自動・確認なし） |

codex 委譲は **ユーザー明示シグナル時のみ・要確認**、下位モデル委譲は **メインモデル判断・自動** という非対称な運用。
