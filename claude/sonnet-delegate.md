# Sonnet 委譲の運用ルール

**目的**: Opus 4.7 のトークン消費量を削減する。実装作業を Sonnet 4.6 (`Agent` + `subagent_type: "general-purpose"` + `model: "sonnet"`) に委譲し、Opus は計画・判断・最終確認に専念する。

## 自動委譲の判断（Opus 側で判定。確認不要）

ユーザーから依頼を受けた直後、**実装に着手する前** に以下の基準で判定する。タイミングが重要 — Opus がファイルを大量に読んでしまった後では、トークンは既に消費済みで委譲効果が薄い。

### 委譲する（Sonnet に投げる）

以下のいずれかに該当し、かつ「設計判断より作業量が支配的」なタスク：

- **複数ファイルにまたがる機械的な編集**: rename、一括置換、API シグネチャ変更の伝播
- **パターン適用 / ボイラープレート**: 既存モジュールに倣った新規スキャフォールド、CRUD、テスト雛形
- **マイグレーション系**: フォーマット変換、import path 一括書き換え、依存ライブラリの API 移行
- **読み込み量が多いタスク**: 多数のファイルを読まないと判断できないが、判断自体は単純なもの（例: 「全テストファイルから deprecated な API 呼び出しを探して直す」）

### 委譲しない（Opus 自身で対応）

- **設計・アーキテクチャ判断**: 抽象化、モジュール分割、API 設計
- **デバッグ・原因調査**: 結果を見ながら方針を変える対話的な作業
- **コードレビュー・批評**: Opus の判断品質が必要
- **小規模な単発編集**: 1〜2 ファイルの軽微な変更（オーバーヘッドが利得を上回る）
- **既に Opus が大量にコンテキストをロード済みのタスク**: トークンは既に支払い済みなので委譲利得が小さい
- **ユーザーが Opus の応答を期待していると明示しているもの**: 「お前が考えて」「Opus に直接聞いている」等

## 委譲フロー

1. **ユーザーへの通知**: 委譲する旨を一言だけ伝える（確認は取らない）。例: 「実装は Sonnet に委譲します」
2. **`Agent` ツールを呼ぶ**:
   ```javascript
   Agent({
     subagent_type: "general-purpose",
     model: "sonnet",
     description: "<3-5 word task>",
     prompt: "<self-contained briefing>",
   });
   ```
3. **プロンプト設計**（トークン効率最優先）:
   - **Sonnet にファイルを読ませる**。Opus 側で読んだ内容を prompt に貼り付けない（二重課金になる）
   - ファイルパス・関数名・変更仕様を **明示的かつ具体的に** 渡す
   - 「報告は変更点の要約のみ。ファイル内容のダンプは不要」と返答形式を指定する
   - プロジェクトの規約は CLAUDE.md から自動的に読まれるので明記不要
4. **結果のハンドリング**: Sonnet からの要約を受け取り、ユーザーに簡潔に伝える。差分の再読み込みは検証が必要なときだけ。

## 安全ルール（委譲しても緩めない）

- **破壊的・不可逆な操作** はユーザー確認を挟む（force push、`git reset --hard`、ファイル一括削除等）。Sonnet に委譲する場合も、Agent プロンプトに「destructive な操作は実行せず Opus に確認を返す」と明記する
- **外部 API 呼び出し・メッセージ送信** は委譲しない（Slack、メール、PR コメント、デプロイ等）
- **シークレット・認証情報を扱うタスク** は委譲しない

## codex 委譲との使い分け

| シグナル | 委譲先 |
|---|---|
| ユーザーが「codex に」「rescue で」と明示 | `codex:codex-rescue` |
| ユーザーが usage limit に近いと発言 | `codex:codex-rescue`（提案あり） |
| 上記以外で、実装量が多く機械的なタスク | `general-purpose` + `model: "sonnet"`（自動・確認なし） |

codex 委譲は **ユーザー明示シグナル時のみ・要確認**、Sonnet 委譲は **Opus 判断・自動** という非対称な運用。
